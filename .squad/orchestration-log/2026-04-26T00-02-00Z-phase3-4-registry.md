# Phase 3–4: CommandRegistry Extraction & Rewiring

**Date:** 2026-04-26  
**Agent:** Coordinator (Phase 3–4 Leader)  
**Session Topic:** Circular dependency elimination  
**Status:** ✅ PHASE 3–4 COMPLETE

## Objective

Extract CommandRegistry.cs to centralize command collections and lookups, rewire AliasManager and CommandPaletteContextMenuFactory to use ICommandLookup interface, and eliminate remaining circular dependencies.

## Changes Implemented

### 1. CommandRegistry Class Created
- **File:** `src/modules/cmdpal/Microsoft.CmdPal.Common/CommandRegistry.cs`
- **Purpose:** Centralized storage and access for:
  - Built-in command collection
  - Pinned commands (dock band)
  - Command ID → command lookups
  - Pin/unpin operations
- **Interface:** Implements `ICommandLookup` with read-only methods:
  - `ICommand GetCommand(string id)`
  - `IEnumerable<ICommand> GetCommands()`
- **Thread Safety:** Maintains internal locks for collection modifications

### 2. ICommandLookup Interface Created
- **File:** `src/modules/cmdpal/Microsoft.CmdPal.Common/ICommandLookup.cs`
- **Purpose:** Read-only command access contract
- **Methods:**
  - `GetCommand(string id)` — retrieve by ID
  - `GetCommands()` — enumerate all available commands
- **Benefits:** Decouples consumers from TopLevelCommandManager; enables interface-based DI

### 3. TopLevelCommandManager Refactored
- **Moved from:** Command collection storage, pin/unpin operations
- **Moved to:** CommandRegistry delegated instance
- **Retained in:** Command loading lifecycle (LoadBuiltinsAsync), extension provider management
- **Property:** Public `ICommandLookup Commands { get; }` exposes registry for consumers

### 4. AliasManager Rewired
- **Before:** Took `TopLevelCommandManager` parameter; accessed TLC.CommandsById and TLC.AllCommandsFlat
- **After:** Takes `ICommandLookup` parameter; calls `GetCommand()` and `GetCommands()`
- **Benefit:** Decoupled from TLC; circular dependency eliminated
- **Circular Dependency Broken:** AliasManager no longer requires full TLC instance

### 5. CommandPaletteContextMenuFactory Rewired
- **Before:** Took `TopLevelCommandManager` parameter; PinToCommand accessed TLC.CommandsById
- **After:** Takes `ICommandLookup` parameter; uses interface methods only
- **Benefit:** Cleaner separation; factory doesn't need full TLC
- **Circular Dependency Broken:** ContextMenuFactory no longer creates tight coupling with TLC

### 6. DI Setup Updated
- **File:** `src/modules/cmdpal/Microsoft.CmdPal.UI/App.xaml.cs`
- **Changes:**
  1. Register `CommandRegistry` singleton
  2. Register `ICommandLookup` → `CommandRegistry.Instance` factory
  3. Update AliasManager registration: inject `ICommandLookup` instead of TopLevelCommandManager
  4. Update CommandPaletteContextMenuFactory registration: inject `ICommandLookup` instead of TopLevelCommandManager
- **Order:** CommandRegistry registered before classes that depend on ICommandLookup

## Circular Dependency Resolution

**Original 3 Cycles:**
1. TopLevelCommandManager ↔ AliasManager ↔ ...
2. TopLevelCommandManager ↔ CommandPaletteContextMenuFactory ↔ ...
3. TopLevelCommandManager ↔ (multiple service consumers)

**After Extraction:**
- TopLevelCommandManager exposes `ICommandLookup Commands`
- AliasManager depends on `ICommandLookup`, not TopLevelCommandManager
- CommandPaletteContextMenuFactory depends on `ICommandLookup`, not TopLevelCommandManager
- No circular edges; clean dependency flow through interface
